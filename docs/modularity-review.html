Critical Issues

Issue #1: openclaw → store (Explain trace internals)

Intrusive Low distance High volatility

What's shared: memory.go reads HybridResult.Explain.Sources string values and pattern-matches on "vec:" / "lex:" prefixes to split vector vs. lexical scores for weighted blending.

Why it matters: These string labels are implementation details of store/hybrid.go's RRF fusion. Any change to source labeling, explain trace structure, or new search types (e.g. "hyde:") silently breaks OpenClaw's scoring. Silent semantic failure — no crash, just wrong results.

Fix: Add VectorScore / LexicalScore fields to HybridResult so OpenClaw reads pre-computed component scores instead of parsing source label strings. ~35 lines across 2 files.

Issue #2: openclaw → config (Config mutation)

Intrusive Low distance High volatility

What's shared: sidecar.go directly mutates cfg.Collections map to inject "openclaw-memory", then calls config.SaveFile() + config.SyncToDB().

Why it matters: Multi-agent support means multiple memory directories per agent. The current single-collection mutation pattern requires touching both openclaw/sidecar.go and config/ internals for every change to collection registration.

Fix: Add config.RegisterCollection(db, cfg, cfgPath, name, opts) that encapsulates mutate-save-sync. Provides a natural extension point for multi-agent registration. ~40 lines across 2 files.

Tolerable Debt

Issue #3: config → database (Raw SQL for store tables)

Intrusive Low distance Low volatility

config/collections.go and config/context.go execute raw SQL against store_collections and store_config tables — tables whose schema is defined in store/schema.go.

Why tolerable: Schema is frozen for TypeScript interchangeability. Revisit if TS compatibility is dropped.

Issue #4: mcp/tools.go → database (Raw SQL bypass)

Intrusive Low distance Low volatility

statusHandler runs direct SELECT COUNT(*) queries instead of calling store.GetStatus(). Minor — replace when convenient.

Issue #5: CLI ↔ MCP orchestration duplication

CLI commands and MCP tool handlers duplicate identical store call sequences for get, multi-get, and partially query. Cohesion issue, not coupling. Monitor for drift bugs.

Well-Balanced Integrations

store → provider (Interface contracts) ✓

Contract Low distance High volatility

Store calls Embedder.Embed(), Reranker.Rerank(), Generator.Generate() through clean interfaces. The right abstraction level — swapping local/remote backends doesn't touch search logic.

format → store (Read-only value types) ✓

Functional Low distance Low volatility

Pure rendering functions. No side effects, no implicit knowledge.

main.go orchestration ✓

Functional Low distance Low volatility

Clean pipeline: config.Load() → db.Open() → store.Init() → provider.New*(). Appropriate for an entrypoint.

db package isolation ✓

db.Open() returns *sql.DB and nothing else. Zero business domain knowledge. Minimal contract coupling.
